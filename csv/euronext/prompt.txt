Why am I getting so strange return JSON?
What are the data fields 'cv', 'iv', 's' mean?
Are values compressed?
======================================================
The data is **not compressed — it's encrypted**. Euronext added client-side encryption as an anti-scraping measure around April 2nd.

The three fields are standard **CryptoJS AES encryption** output:

| Field | Meaning | Details |
|-------|---------|---------|
| `ct` | **ciphertext** | The actual encrypted payload (base64-encoded). This contains your original JSON with `rows`, `count`, etc. |
| `iv` | **initialization vector** | 16-byte hex string (128-bit IV) used for AES-CBC mode |
| `s` | **salt** | 8-byte hex string (64-bit salt) used for key derivation |

**How it works:** The Euronext website JavaScript includes a passphrase/key. When the browser receives this encrypted JSON, client-side JS calls something like `CryptoJS.AES.decrypt(response, passphrase)` to recover the original plaintext JSON. The salt (`s`) is used with the passphrase via OpenSSL's `EVP_BytesToKey` (PBKDF) to derive the actual AES key and IV.

**Why you see it now:** When you visit the URL directly in the browser address bar, no page JavaScript runs, so you get the raw encrypted blob. When browsing the actual Euronext page, their JS decrypts it transparently before rendering.

**To decrypt it** you'd need to:
1. Find the decryption passphrase embedded in Euronext's JavaScript bundles (inspect their `.js` files loaded on the intraday price page)
2. Use that passphrase with CryptoJS-compatible AES decryption (OpenSSL EVP_BytesToKey key derivation with the salt, then AES-CBC decrypt with the derived key and IV)

This is a deliberate anti-scraping/anti-bot protection. The passphrase may rotate periodically, making automated access harder. Your existing Go code that fetches this endpoint will need to be updated to handle decryption if you want to continue using it.
